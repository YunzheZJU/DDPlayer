设计决策
========

为什么叫DDMusic？和DDPlayer是什么关系？
--------------------------------------
DDMusic是Anisong企划中所有产品的整体名称。

DDPlayer是DDMusic中前端工程的产物。

DD的含义
--------
* 不管是谁都最喜欢了
* 男大学生

企划的内容
----------
依托校园网内网的可靠性和速度，以及资源上传者们的付出，为校内ACG爱好者们提供体验好，质量高的在线音频点播服务。

提供的音频内容为与ACG有关的所有音频资源（动画OP/ED/OST/角色歌、声优偶像、动画歌手、声优广播和广播剧）。

主题色
------
使用大红色（hsl(5, 80, 82)）作为主题色，衍生出同色系的不同色彩用于界面设计。

信息架构与快速启动区
--------------------
我想创建一种较为扁平的信息架构，在根节点的下方，有“发现”、“歌手”、“专辑”、“歌单”、“歌曲”和“搜索”等页面。

前四个和“搜索”在内容区顶部提供，“歌曲”即单曲页，入口也设置在顶部（窗口的右上角专辑封面）。

即所有主入口都安排在窗口顶部区域。

快速启动区的作用是将六大主页面的常用子页面提取出来提供给一个快速入口。

以歌手为主轴的内容结构
----------------------
现实中的音乐生产过程是专辑-歌曲-歌手，而观众直观感受到的是歌手的魅力。

因此DDPlayer将歌手摆在首位，构建歌手-歌曲-专辑/歌单的内容结构。

栏目
----
颠覆流行桌面端音乐播放器的相似设计，构建出内容区、歌单区和控制区的纵向三栏设计，重排功能组件，探索新颖且有效的交互方式。

进度条为顶部交互轴线，其末端为专辑封面，专辑封面引出另一条交互轴线，配合主题色将用户的注意力吸引到重排过后的控制区。

按钮哪去了
----------
重排了，有的换了位置，有的被合并，但是功能没有被移除，我为它们提供了至少1种替代入口以保证可访问性。

没有顶部区
----------
传统顶部区的Logo、搜索、用户信息、设置和窗口按键没有消失，他们被重排到了其他位置。

右上角专辑封面
--------------
右上角是仅次于左上角的关注点，将封面摆放在这个位置并且突出显示，让用户第一眼就能想起自己在听什么歌。（图片更容易记忆）

同时这是单曲页面的入口，将它做得醒目，保证用户可以快速点击。

当前播放的歌曲信息就在封面的左侧，也很容易获取。当然更推荐单击封面进入单曲页面浏览。

进度条
------
做了一个将近全幅的进度条摆放在应用顶部，让用户直观快速地明白当前播放进度。

进度条的“存在感”是靠占据的尺寸来体现的，更长的进度条也提升了跳转精度。

由于界面采用纵向三栏设计，顶部和底部是仅有的可以容纳全幅宽度的位置。

考虑过做纵向的进度条，但是过于违反直觉。

播放/暂停
---------
做得很大，因为我认为这是一个非常常用的功能，更大的尺寸赋予它更高的关注度。

并且我在左侧为它制作了一个凸起的装饰（已弃用），也是为了在用户需要时快速吸引注意力。

我为它添加了“滑动切歌”的功能，意在尝试新的交互方式。

手势是仅次于直接点击的交互方式。

在收起切歌控制之后依旧可以通过滑动的手势来切换歌曲，既节省了空间又不缺失功能，与手势有很好的语义结合。

切歌控制
--------
（已弃用）“上一曲”和“下一曲”或许不是那么常用的功能（未验证），所以我将它摆放在暂停/播放按钮的下方并提供“收起”功能。

现在切歌控制被安排在滑动切歌中和模式按钮的两侧。

收起按钮
--------
（已弃用）灵感来自Maya工具栏，收起和展开不常用的按钮可以节省空间，避免它们拖延用户获取目标的速度，在需要时也可以快速展开按钮。

交互轴线断开了吗
----------------
没有。在主题色的暗示之下，用户会完整地扫过右侧的控制区。控制区四大内容：封面、控制按钮组、播放按钮和用户信息。

控制按钮组的增减按钮
--------------
控制按钮组的增减按钮主要目的是在精简掉“上一曲下一曲”按钮之后提供一个替代入口以免功能缺失。

为了形式统一，在音量和歌词两侧也都增加了小按钮。

同时，我设计了滑动主按钮来选中小按钮的操作，提供小屏幕上的易用性。

歌曲区设计
----------
歌曲区的设计借鉴了网易云音乐歌曲区信息排版，三块内容分别为封面、歌名/歌手/时间、快捷功能（收藏、分享和下载）。

前两者设计为单曲页面入口。

比网易云音乐（其他播放器没有歌曲区）多出了“时间”和“下载”。

“时间”摆放在进度条末端下方，“下载”在悬停“分享”之后显示。

快速启动区设计
--------------
精简了不需要的页面入口（本地音乐、音乐视频等），保留“播放队列”、“发现新音乐”、“收藏内容”

歌单区设计
----------
精简掉了常用的“我创建的歌单”和“我收藏的歌单”，“歌单”用不同的图标来区分创建者。

创建新歌单
----------
在歌单区内容块的末尾追加“创建新歌单”图标（不会因歌单过多而被隐藏）。

常见的做法是放在“我创建的歌单”右侧，但因为我精简掉了常用的“我创建的歌单”和“我收藏的歌单”所以放在这里。

内容区
----------
内容区是以一个固定在顶部的标签栏和多个动态页面构成的页面。

内容区提供推荐（某种形式的首页，暂定）、歌手筛选排序、专辑排行查看、歌单浏览和搜索功能。

搜索会稍稍改变内容区顶部布局。

内容区悬浮控件
--------------
没有地方放了就做了悬浮按钮，当页面滚动一定距离时自动隐藏（靠近显示），向上回滚时浮出。

提供前进、后退和刷新功能（浏览器本身也提供）。

内容区标签设计
--------------
内容区标签栏共有“发现”、“歌手”、“专辑”和“歌单”四个标签，当内容区内容符合某个标签时这个标签会高亮。

在任意状态下激活某个标签，内容区会跳转至相应的标签页：

* “发现”标签提供推荐内容，同时作为应用启动首页。

* “歌手”标签提供按首字母和性别筛选排序歌手，具有“热门”选项以快速筛选。“歌手”标签也包含了“我加推的歌手”的入口。

* “专辑”标签提供按发行日期筛选排序专辑，具有“热门”选项以快速筛选。“专辑”标签也包含了“我收藏的专辑”的入口。

* “歌单”标签包含播放队列、“我喜欢的音乐”和我创建的歌单的入口。

搜索
----
在内容区右上方建立了一个搜索框，激活搜索框之后一个绝对定位的搜索提示将被弹出以用于显示搜索内容。

若将搜索作为歌单区单独一行，将违反“点击歌单区某一行内容之后内容区会发生跳转”的直觉。

搜索范围依次为“歌手”、“专辑”、“歌曲”和“歌单”。“歌单”包含“我喜欢的音乐”和“我创建的歌单”。

搜索结果
--------
搜索结果按照类型的不同使用了不同的默认信息展现形式。

歌手和专辑使用了略缩图的展现形式，符合用户浏览的Z型模式，横向浏览略缩图，若有感兴趣的则停下，否则扫读。悬停显示详细信息。

歌曲和歌单使用了列表的展现形式，符合用户浏览的F型模式，一旦名称不匹配/不感兴趣则无需继续看这一行。

快捷键“/”
----------
借鉴GitHub快捷键，快速激活搜索框。

搜索关键词高亮
--------------
高亮颜色取了与红色相对的蓝色，我不太明白原理，只是借鉴了众多播放器的做法。

尽管它会影响页面色彩平衡，但是有了关键词高亮，用户才能知道搜索结果的匹配程度和匹配位置，这对凭印象来搜索的用户来说很重要。

歌单列表
--------
最右侧放了操作常用按钮和“更多”按钮，点击“更多”按钮可以看到完整的操作选项，和在该项单击右键的效果是一样的。

单曲页
------
单曲页分为图像区、信息区、操作区和推荐区。

图像区包含正方形的专辑封面、环状的进度指示器和动态生成的频谱可视化图形。

信息区包含必要的歌曲信息，也为“标签”预留了位置。

操作区包含“喜欢”、“添加”、“下载”和“分享”，样式借鉴了bilibili移动端番剧详情页。

推荐区有“相关歌曲”和“喜欢这首歌的人”两个板块，各能容纳2和12条相关内容。

单曲页使用了高斯模糊的专辑封面来等比例铺满背景。
